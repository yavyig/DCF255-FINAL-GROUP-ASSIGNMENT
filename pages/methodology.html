IMAPs Compatibility

IMAP4 is a new implementation derived from IMAP2. In fact, the latter is designed to be upwards compatible to the former despite the fact that some early protocols in IMAP2 have been obsoleted, such as commands, responses, and data formats, which describes in RFC 1730.

IMAP2 was made for the purpose of supporting the older RFC 822 standard, which defines an electronic message format consisting of header fields and an optional message body. As a result, several fetch items (also known as “citation linker”) in IMAP incorporate RFC 822 in their name.


Protocol Inclusivity

There are several protocol conventions in IMAP. They provided the concepts to the IMAP protocol which, however, doesn’t mean they are strictly followed but reflect generally applicable practice. When implementing, it is important to know their conventions to avoid conflicts. For instance, “&” may not used as a hierarchy delimiter since it conflicts with the Mail International Naming Convention, and other conventions use this symbol will also be affected.


TCP/IP Link Port

IMAP4 retrieves email messages from a mail server over a TCP/IP connection. When TCP is used, an IMAP4 server listens on port 143; if SSL/TLS, IMAP is assigned the port 993.



COMMAND AND RESPONSES

To establish a IMAP4rev1 connection between client side and server side, an initial greeting from the server is needed for the interactions. These client/server interactions consist of 3 elements:

  1. Client command
  2. Server data
  3. A server completion result response

The form of lines is the way used for all interactions to transmit. Each line ends with CRLF and has prefix of an identifier; generally, a short alphanumeric string like A0001, A0002. The client is the one who start building an operation with a string command.


MESSAGE ATTRIBUTES

Messages in IMAP4 are access by one of two numbers: (1) the unique identifier number or (2) the message sequence number.


The Unique Identifier Number (UID)

The UID message attribute is a 32-bit value assigned to each message used with the unique identifier validity value to form a 64-bit value where the reference to any other message in the mailbox or any newly created mailbox with the same name are strictly prohibited. Unique identifiers MUST be assigned in an ascending order that is, when messages are added to the mailbox, they are subsequently assigned UID with incremental values. If the physical messages are rearranged by an external non-IMAP agent, the re-generation of UID will be required; if not, the ascending rule will be broken. The unique identifier of a message must not change during the session, and never change between sessions.




Message Sequence Number Message Attribute

A relative position from 1 to the number of messages in the mailbox. This position MUST be ordered by ascending unique identifier. As each new message is added, it is assigned a message sequence number that is 1 higher than the number of messages in the mailbox before that new message was added.

Message sequence numbers can be reassigned during the session. For example, when a message is permanently removed (expunged) from the mailbox, the message sequence number for all subsequent messages is decremented. The number of messages in the mailbox is also decremented. Similarly, a new message can be assigned a message sequence number that was once held by some other message prior to an expunge.


STATE AND FLOW DIAGRAM

Once the connection between client and server is established, an IMAP4rev1 connection is in one of four states. The initial state is identified in the server greeting. Most commands are only valid in certain states. It is a protocol error for the client to attempt a command while the connection is in an inappropriate state, and the server will respond with a BAD or NO (depending upon server implementation) command completion result.

Authenticated State

In the authenticated state, the client is authenticated and MUST select a mailbox to access before commands that affect messages will be permitted. This state is entered when a pre-authenticated connection starts, when acceptable authentication credentials have been provided, after an error in selecting a mailbox, or after a successful CLOSE command.

Selected State

In a selected state, a mailbox has been selected to access. This state is entered when a mailbox has been successfully selected.



Logout State

Logout state indicates that the connection is being terminated. This state is triggered by the client via a logout command, or or by unilateral action on the part of eitherthe client or server.
